# Read-only Candidate Diagnostics Integration Design

Date: 2026-06-09
Author: Antigravity Agent
Task: Task 44 — Read-only candidate diagnostics integration design
Scope: Governance design note only. No product code changes permitted in this task.
Product baseline: `7b64ba53afb43ca43727e3a4c028fee93a2cc23c`

## Purpose
This document outlines how the newly implemented primitive evidence candidate models and extractor can be integrated into the existing `inspect_pdf` diagnostic payload. The goal is to provide read-only, geometry-preserving supplementary candidate diagnostics without modifying the core evidence-gathering pipeline or introducing semantic heuristics.

## Verified Product State
The candidate extraction boundary is complete:
- `PrimitiveGeometryEvidence` and `XAlignedPrimitiveClusterEvidence` are fully defined.
- `LeftMarginPrimitiveCandidate` and `XAlignedPrimitiveClusterCandidate` models exist and enforce strict validation against semantic leakage.
- `PdfGeometryCandidateExtractor` accurately maps evidence to candidates while inheriting context-level `page_index`, `system_index`, and `staff_index`.

## Existing Candidate Boundary
The candidate boundary strictly isolates geometric evidence from meaning. The models only record bounding boxes (`x0`, `y0`, `x1`, `y1`), morphological kinds, font data, and specific sources (`left_margin`, `x_aligned_cluster`). Candidate derivation from aggregate counts is expressly prohibited.

## Proposed Read-only Diagnostics Attachment
A future implementation should add supplementary candidate diagnostics fields to the `NotationStaffDiagnostics` model. These fields will house the instantiated candidate models generated by the `PdfGeometryCandidateExtractor`. The new fields must be strictly optional or default to `None` to prevent breaking existing schema contracts. A distinction between `None` and `[]` will be enforced to accurately reflect extraction states without fabricating data.

## Invocation Plan
In a future product task, `build_notation_diagnostics` inside `src/score2gp/pdf_staff_notation_diagnostics.py` would instantiate the `PdfGeometryCandidateExtractor`.
- **Context:** The `page_index`, `system_index`, and `staff_index` are already computed during the iteration over `notation_groups`. These variables must be passed directly into the extractor.
- **Left-margin:** If `left_margin_diags` is successfully generated and contains an `evidence` array, `extract_left_margin_candidates` should be called using `left_margin_diags.evidence` and the current staff context.
- **X-aligned clusters:** If `clustering_diags` is successfully generated and contains an `evidence` array, `extract_x_aligned_cluster_candidates` should be called using `clustering_diags.evidence` and the current staff context.

## Proposed Payload Shape
The `NotationStaffDiagnostics` model schema (or the appropriate product model, which must be verified during Task 45) should be extended with two new read-only optional fields:
- `left_margin_candidates: list[LeftMarginPrimitiveCandidate] | None = None`
- `x_aligned_cluster_candidates: list[XAlignedPrimitiveClusterCandidate] | None = None`

This ensures that any output is strictly an extension and supplementary, residing alongside the existing `morphology` and `clustering` models. The schema choice must be consistent: a value of `None` means "not run or evidence unavailable", whereas an empty array `[]` means "run but no candidates found". If evidence arrays are missing entirely, the future implementation must default to `None`.

## Preservation Requirements
The future integration must ensure that:
- Candidate output preserves the source geometry bounds exactly as found in the raw evidence array.
- Candidate output preserves input order.
- Existing aggregate morphology and clustering diagnostics remain completely unchanged.
- The `PdfStaffNotationGeometryDiagnostics` payload structure remains intact, simply carrying the additional candidates as read-only appendices on each staff.

## Backward Compatibility Requirements
Candidate diagnostics must be strictly optional, defaulting to `None`, to protect backward compatibility. Existing consumers of the JSON report must not fail due to the sudden appearance of these new schema keys.

## Validation Required Before Any Product Implementation
Before any code is committed, the following validations must pass:
- All semantic gates (`test_pdf_candidate_semantic_gate.py`) must continue to verify the absence of musical semantics.
- Pytest must confirm no existing fixtures break when the payload size and structure expand.

## Privacy and Artifact Constraints
No integration work shall involve committing private input PDFs, reference screenshots, GP files, or debug artifacts to the product repository.

## Explicit Non-goals
- Do not infer meaning.
- Do not recognise musical objects.
- Do not classify notes.
- Do not convert candidates to ScoreIR.
- Do not use candidate output to replace primitive diagnostics.
- Do not generate snapshots now.
- Do not modify product schema in this task.
- Do not implement integration in this task.

## Stop Conditions for Future Implementation
Any future product implementation task must stop immediately if:
- It requires generating candidate bounds that do not exist in the source evidence.
- It attempts to group candidates into musical objects (e.g. forming "notes" or "chords").
- It requires altering the `extract_left_margin_candidates` or `extract_x_aligned_cluster_candidates` contracts.
